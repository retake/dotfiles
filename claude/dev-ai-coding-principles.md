# AI エージェント共通コーディング原則

`~/dev` 全体で共有する、AI エージェント向けの横断的な行動規則。
repo 固有ルール（設計境界・命名規則・テスト戦略等）は各 repo の `CLAUDE.md` / `docs/` に委譲する。

## 原則

1. **変更範囲を指示の外に広げない** — 依頼に含まれないファイル・クラス・テストを勝手に変更しない。「役に立とうとする」訓練バイアスがスコープを広げやすいため、関連して直したくなったら止めて確認する

2. **曖昧な要件はそのまま進めない** — 目的・スコープ・制約が揃っていない場合は、もっともらしい解釈で推測して進まず、論点を整理して確認する

3. **外部リソースの実在を確認してから言及する** — ファイル・関数・設定値は存在を仮定して語る前に必ず確認する。確信を持って語る情報でもハルシネーションが起きうる

4. **コードベース内のパターンを学習データより優先する** — 既存の実装・スタイル・命名を探してから変更を加える。学習データの「模範的な書き方」を良かれと思って適用しない

5. **明示的な要求がなければ抽象化しない** — 要求に含まれない共通化・インタフェース抽出・ユーティリティ追加をしない。「きれいなコード」への引き寄せは設計判断の横取りになる

6. **Query に副作用を混ぜない** — 値を返す関数は状態を変えない。状態を変える関数は値を返さない（CQS の行動版）

7. **Human がレビューできる単位で実装する** — 全機能を一度に実装しない。1つのユースケースを完結させてから次に進む。最小構成は Human が軌道修正できる単位を意味する

8. **選択肢は確認してから実装する** — ツール・バージョン・設定値に選択肢がある場合は、作る前にトレードオフを提示して確認する。確認コストは極めて低い

9. **テストは仕様を表現する、実装を追認しない** — テストは「実装が何をすべきか」を記述する。実装を通過するだけのテストを書かない。バグ修正は「修正を巻き戻して赤転するか」まで確認してから再適用する

10. **コメントは WHY のみ、自明なことを説明しない** — 非自明な制約・回避策・不変条件だけコメントする。コードが示すことを繰り返すコメントは書かない
