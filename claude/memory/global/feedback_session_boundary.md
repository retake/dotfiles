セッションは「1 セッション = 1 意図スレッド」で区切る。意図が変わったら別セッションに切る。

**Why:** 長いセッションは cache は温かいが、無関係な話題が混ざると精度が落ち、/retro も冗長になる。短く切れば 1 turn あたりのコストも下がり、振り返りが濃く書ける。ユーザーが 2026-04-13 に自分で言語化した運用原則。

**How to apply:**

閉じ時（目安）:
- commit + /retro が終わった自然な checkpoint
- 扱う対象が切り替わる時（実装 → ドキュメント整理 → メタ作業などの跨ぎ）
- 「次に何をするか」で迷い始めた時（その迷いは context 汚染のサイン）
- 文脈が 50% を超えた時（cache は効くが精度が落ちる）

続けてよい時:
- audit → 実装 → audit の連鎖のように作業が次を自然に呼んでいる時
- 直前の作業への小さな修正・連続リファクタ
- キャッシュが温かくタスクが前進している間

閉じる前のチェック:
1. 未 commit 差分はあるか? → あればコミット or 破棄を決める
2. 今日「未来の自分が同じ迷いをしないためのルール」が出たか? → 出たなら persist
3. どの層に書くか（memory / root / dev / project CLAUDE.md / settings.json）を明示する

2 問目が「なし」なら persist せず閉じる。何も出ない日も多く、毎回ルール追加するとノイズになる。
