# orchestrate・サブエージェント・スキル設計の運用ガイド

orchestrate スキル実行時やスキル設計時に参照する。日常の実装セッションでは不要。

---

## サブエージェントのガードレール

- **ゲートポイント**: 読み取り・編集・コミットは自動。push・外部送信は人間承認。認証情報・インフラは禁止
- **ループ指示**: 成功条件・上限・エスカレーション条件を必ず含める
- **教訓注入**: Agentツール呼び出し前に `~/retrospectives/_index.md` を読み、プロンプトに明示的に含める（PreToolUseフックのadditionalContextはサブエージェントに届かないため）
- **教訓記録**: ループ完了後、学びがあれば「memoryに記録しますか？」と確認する（承認後のみ記録）
- **allowリスト**: サブエージェントに自動実行させるコマンドは事前に `settings.json` の `allow` に追加する
- **手動実装時のドキュメント同期**: orchestrate 非経由の手動実装でも、実装直後に design-summary.md・traceability.md を更新する。**コミット前に `/sync` スキルを実行**してコードとドキュメントの乖離を自動検出・補完するのが最も確実（乖離なしの判定が出れば手動追記は不要）。`/audit-handoffs` → `/archive-handoffs` の 2 段運用で残課題の視認性を維持する
- **orchestrate を「判断タスク」に使う運用**: 要件レベル意思決定にも使える。task-state.md で FR-4 以降を SKIP、normal モードで FR-3 architect 停止ゲートを活用する
- **累積台帳型ドキュメントの上書き事故防止**: architect 呼び出し時は「既存を Read → 末尾に `---` 区切り + `# 追補: TASK-ID` セクションとして Edit で追加」を明示する。既存行は変更/削除しない制約も同時に書く
- **orchestrate タスクサイズの確定基準**: 確定サイズは「本タスクで新規追加/大幅改修する REQ 数」を基準にする。Orchestrator 裁量で維持した場合は task-state.md の「確定サイズ」欄に判断理由を明記する

## スキル設計原則

- **途中停止前提のリレー設計**: 複合処理スキルは初回から最後まで走り切らない。ステップの境界で handoff 起票を用意し、途中停止→次セッション引き継ぎを織り込む
- **対話 UX は「候補先出し→採否確認」**: Claude 側が先に候補を列挙し、ユーザーは採否のみ答える形式にする
- **クロスプロジェクトは技術スタック非依存で書く**: パスをハードコードしない。自動探索または CLAUDE.md でのパス宣言参照で解消する
- **スキル間の情報連携はファイル経由で構造化する**: 中間ファイル経由で参照する構造にする
- **スキルは自己言及禁止**: 処理手順はステップ番号で明示し、条件分岐は if-else 形式で書き下す
