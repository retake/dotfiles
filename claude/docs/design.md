# Claude Code 自律エージェント活用フロー 設計ドキュメント

## 1. エージェント構成

### 方式：オーケストレーター型

理由：FR-2・FR-3・FR-5/FR-6 の各所で人間介入・承認待ちが発生する。
パイプライン型より条件分岐・状態管理の集中制御が容易なため。

### 各エージェントの役割

| エージェント | 担当FR | 役割 |
|---|---|---|
| Orchestrator | FR-1, FR-2, FR-9, FR-10 | 入力解釈・確認停止制御・状態管理・エスカレーション記録 |
| Architect | FR-3 | モジュール構成・インタフェース定義・ディレクトリ構造・命名規則の生成とサマリ出力 |
| Implementer | FR-4 | コード生成・要件IDコメント付与 |
| Tester | FR-5 | テスト実装・実行・自律修正 |
| Linter | FR-6 | lint実行・自律修正 |
| Reviewer | FR-7, FR-8 | 4軸レビュー・限定的なコード品質の自動修正・成果物出力 |

### エージェント間のデータ受け渡し

- ファイル経由（コンテキスト肥大化防止・再開可能性確保）
- Orchestratorがサブエージェント呼び出し時に `.claude/task-state.md` の現フェーズと成果物パスをプロンプトに明示する
- サブエージェントは完了時に成果物ファイルを所定パスに書き込み、終了ステータスをOrchestratorに返す

## 2. タスクフロー

```
ユーザー入力
    │
    ▼
[FR-1] 入力解釈・必須項目確認・lint設定確認・タスクサイズ自動判定
    │
    ├─ 必須項目未記載 → [FR-9] 確認停止 → 再入力 → FR-1へ戻る
    │
    ├─ 技術スタック未指定 → [FR-2] 技術選定提示（ライブラリカテゴリ含む）→ 確認停止
    │                              └─ 承認 → FR-3へ
    │
    └─ 技術スタック指定済み → FR-2スキップ → FR-3へ
    │
    ▼
[FR-3] 設計（モジュール構成・インタフェース・ディレクトリ構造・命名規則）
    │
    └─ 設計サマリ出力 → [FR-9] 確認停止（承認待ち）
                              └─ 承認 → FR-4へ
    │
    ▼
[FR-4] 実装（コード生成・要件IDコメント付与）
    │
    ▼
[FR-5] テスト実装・実行（上限回数は規模・エラー種別で決定）
    │
    ├─ 成功 → FR-6へ
    └─ 失敗 → 自律修正（上限内）→ 再実行
          └─ 上限超過 → [FR-9] エスカレーション停止
    │
    ▼
[FR-6] lint実行（上限回数は規模・エラー種別で決定）
    │
    ├─ 成功 → FR-7へ
    └─ 失敗 → 自律修正（上限内）→ 再実行
          └─ 上限超過 → [FR-9] エスカレーション停止
    │
    ▼
[FR-7] レビュー（4軸：要件整合・トレーサビリティ・コード品質・セキュリティ）
    │
    ├─ セキュリティ問題検出 → [FR-9] 確認停止（自動修正禁止）
    ├─ インタフェース変更が必要 → [FR-9] 確認停止 → FR-4やり直し（上限1回）
    │                                  └─ 2回目のエスカレーション → 中止
    └─ コード品質指摘 → ロジック上の不具合のみ自動修正
                          スタイル修正（命名・フォーマット等）はLinterの責務のため行わない
                          自動修正した場合 → FR-5 → FR-6 を再実行
    │
    ▼
[FR-8] 成果物出力（コード・コメント・traceability.md・完了サマリ）
    │
    ▼
[FR-10] task-state.mdに完了記録 → ターミナルへ完了通知
```

### 確認停止が発生するポイント（7箇所）

1. FR-1後：必須項目未記載
2. FR-2後：技術選定の承認待ち（スタック未指定時のみ）
3. FR-3後：設計サマリの承認待ち
4. FR-5後：テスト失敗が上限超過
5. FR-6後：lint失敗が上限超過
6. FR-7中：セキュリティ問題を検出
7. FR-7中：インタフェース変更が必要（FR-4やり直し、上限1回）

### 確認停止・エスカレーションの出力フォーマット

**承認待ち：**
```
[STOP] フェーズ: FR-3 設計承認待ち
理由: 設計サマリの人間承認が必要です
確認ファイル: docs/design-summary.md

再開方法:
  承認: 「承認」または「OK」と入力
  修正あり: 修正内容を入力
  中止: 「中止」と入力
```

**エスカレーション：**
```
[ESCALATION] フェーズ: FR-5 テスト失敗（上限超過）
停止箇所: src/auth/token.ts:42
状況:
  試行1: 型アサーション追加 → 失敗
  試行2: 型定義修正 → 失敗
  試行3: インタフェース変更 → 失敗
選択肢:
  A) 手動修正後「再開」と入力
  B) スキップして「スキップ」と入力（レビュー指摘として記録）
  C) 「中止」と入力
記録場所: .claude/task-state.md
```

## 3. settings.json の設計方針

### permissionsのallowリスト

自動実行を許可するコマンド（プロジェクト技術スタック確定後に追記）：

```json
{
  "permissions": {
    "allow": [
      "Bash(find:*)",
      "Bash(ls:*)",
      "Bash(grep:*)",
      "Bash(git status:*)",
      "Bash(git diff:*)",
      "Bash(git log:*)",
      "Bash(git add:*)",
      "Bash(git commit:*)",
      "Bash(date:*)",
      "Write",
      "Edit"
    ],
    "deny": [
      "Bash(git push:*)",
      "Bash(rm -rf:*)",
      "Bash(ssh:*)",
      "Bash(npm publish:*)"
    ]
  }
}
```

※ git add / git commit はOrchestratorのみが使用する。サブエージェントのプロンプトで明示的に禁止する（allowリストには残すが設計上の制約として管理）。

※ テスト・lint・ビルドコマンド（npm test、pytest等）はプロジェクトの技術スタック確定後にallowリストへ追加する。
※ 過去の教訓：委譲前にallowリストへ追加しておく。FR-2承認時またはFR-2スキップ時（スタック指定済み）はFR-1完了後にOrchestratorが確認停止し、allowリスト更新を促してから次フェーズへ進む。

### hooksの設計

hooksはコマンドの許可/拒否の静的ルールに限定する。カウント管理・フラグ管理はOrchestratorの責務（セクション5参照）。

**PreToolUse（ツール呼び出し前）：**
- ファイルパスがプロジェクトディレクトリ外の場合はブロック（NFR-1）
- `~/.ssh`・認証情報ファイルへのアクセスをブロック

**PostToolUse（ツール呼び出し後）：**
- `{project_root}/.claude/agent-log.md` にツール呼び出し結果を追記（書き込み専用・エージェントは読まない）
- パスはtask-state.mdの `project_root` フィールドから絶対パスで解決する（フックのcwdはプロジェクトルートと一致する保証がないため相対パス不可）

### サブエージェントのガードレール

- Agentツール呼び出し前に `~/dev-log/retrospectives/_index.md` を読み、教訓をサブエージェントのプロンプトに明示する
- 各サブエージェントのプロンプトに「スコープ外の判断が必要な場合は人間に確認して停止」を必ず記載する
- サブエージェントは担当FR以外の操作を行わない（役割の境界をプロンプトで明示）

### `.claude/task-state.md` のフォーマット

Orchestratorが起動時に必ずこのファイルを読み、`status` に応じて処理を分岐する。

| status | 意味 | Orchestratorの動作 |
|---|---|---|
| running | 実行中 | 現在のphaseから継続 |
| waiting_approval | 承認待ち | 承認確認メッセージを出力して停止 |
| escalated | エスカレーション | 停止箇所・理由を出力して人間の指示を待つ |
| done | 完了 | 完了済みタスクとして扱う |

```markdown
# Task State

## メタ情報
- タスクID: TASK-YYYYMMDD-NNN
- 入力日時: YYYY-MM-DD HH:MM
- 目的: （入力テンプレートの目的）
- 技術スタック: （FR-2で確定したスタック）
- 承認済みライブラリカテゴリ: テスト用・型定義・開発ユーティリティ
```

## 6. ツール呼び出し回数の管理

カウント管理はOrchestratorの責務。PreToolUseフックには任せない。

### タスクサイズの2段階判定

**FR-1（暫定）：要件ID数で判定**

| サイズ | 上限 | 判定基準 |
|---|---|---|
| small | 50回 | 要件IDが1〜2件 |
| medium | 150回（デフォルト） | 要件IDが3〜7件 |
| large | 300回 | 要件IDが8件以上 |

**FR-3完了時（確定）：モジュール数・ファイル数で上書き**

| サイズ | 判定基準 |
|---|---|
| small | 変更ファイル数1〜2、モジュール数1 |
| medium | 変更ファイル数3〜5、モジュール数2〜3 |
| large | 変更ファイル数6以上、モジュール数4以上 |

### カウントの管理方法（フェーズ別固定上限方式）

- 各サブエージェントにはフェーズ別の固定上限を割り当てる（グローバル残余に依存しない）
- 目的の分離：**フェーズ別固定上限 = ループ防止**、**グローバル上限 = コスト観測**
- サブエージェントは割り当て枠内で完結し、完了時に使用回数をtask-state.mdのagent-reportセクションに追記する
- グローバル累計はOrchestratorが各FR完了後に更新する（観測用・超過時は警告）

**フェーズ別固定上限：**

| エージェント | small | medium | large |
|---|---|---|---|
| Architect（FR-3） | 10回 | 20回 | 40回 |
| Implementer（FR-4） | 20回 | 50回 | 100回 |
| Tester（FR-5） | 14回 | 35回 | 70回 |
| Linter（FR-6） | 6回 | 15回 | 30回 |
| Reviewer（FR-7/8） | 10回 | 20回 | 40回 |
| **合計** | **60回** | **140回** | **280回** |

## 7. 自律修正の上限回数

テスト失敗・lint失敗の上限回数はタスクサイズとエラー種別で決定する。

### エラー種別

| 種別 | 内容 | 理由 |
|---|---|---|
| 通常エラー | 型エラー・ロジックエラー・lint指摘 | 自律修正で解決できる可能性が高い |
| 外部依存エラー | ネットワーク・DB・ファイルシステム起因 | 自律修正で解決できる可能性が低い |

### 上限回数の表

| タスクサイズ | 通常エラー | 外部依存エラー |
|---|---|---|
| small | 2回 | 1回 |
| medium | 3回 | 1回 |
| large | 5回 | 1回 |

外部依存エラーはエラーメッセージでネットワーク・タイムアウト・接続拒否等を検知して判別する。

判別キーワード（いずれか含む場合は外部依存エラー、大文字小文字を問わず判別）: `timeout`, `ECONNREFUSED`, `connection refused`, `ENOTFOUND`, `network`

## 7b. FR-4やり直し時の管理

ReviewerがインタフェースESCALATEDを返した場合のFR-4やり直しルール。

### 基本方針

| 項目 | 方針 |
|---|---|
| ツール呼び出し累計 | 引き継ぐ（リセットしない） |
| FR-4〜FR-7のステータス | PENDINGに戻す（完了日時もクリア） |
| FR-5・FR-6の自律修正試行回数 | リセットする（コードが変わるため） |
| FR-4再実行の上限 | 1回（2回目のESCALATEDで中止） |
| 残余枠チェック | 残余 < 最低必要枠のとき確認停止 |

**FR-4 やり直しで必要な追加コスト：**

フェーズ別固定上限方式では各フェーズが独立して実行されるため、「残余枠チェック」は不要。
ただし FR-4再実行回数が上限（1回）に達した場合は中止する（§7b 手順1参照）。

| サイズ | FR-4〜FR-7の1サイクル合計 |
|---|---|
| small  | 20+14+6+10 = 50回 |
| medium | 50+35+15+20 = 120回 |
| large  | 100+70+30+40 = 240回 |

※ task-state.mdへの追加フィールド（`FR-4再実行管理`・`自律修正試行回数`・`エスカレーション連番`）は §4 のフォーマット定義に統合済み。

### やり直し時のOrchestratorの処理手順

1. FR-4再実行回数を確認する
   - すでに1回の場合: [ESCALATION] で中止（設計レベルの問題として人間に委ねる）
2. design-summary.mdの更新確認停止（以下を出力して停止）：
   ```
   [STOP] フェーズ: FR-4 再実行前 設計更新確認
   理由: インタフェース変更が必要です。design-summary.mdを更新してください
   変更箇所: （ReviewerのESCALATED内容を転記）
   ファイル: docs/design-summary.md

   更新後に「承認」と入力してください
   （Implementerは更新済みのdesign-summary.mdを元に再実装します）
   ```
4. task-state.mdを更新する
   - FR-4再実行回数: +1
   - FR-4再実行理由: ReviewerのESCALATED内容
   - FR-4〜FR-7のステータス: PENDING（完了日時もクリア）
   - FR-5試行回数・FR-6試行回数: 0にリセット

## 8. スキルの定義（自動実行と確認の境界）

### スキルに含まれるもの（自動実行対象）

基準：可逆性があり、プロジェクトディレクトリ内で完結し、セキュリティ判断を含まないもの。

- ファイルの読み取り・検索・静的解析
- ファイルの編集・新規作成（プロジェクトディレクトリ内）
- allowリスト記載済みのテスト・lint・ビルドコマンド実行
- git add / git commit（push・force系を除く）
- `.claude/` 配下のファイルへの書き込み

### スキルに含まれないもの（確認が必要）

- git push・デプロイ・外部サービスへの送信
- 認証情報・インフラへのアクセス
- セキュリティ上の設計変更（脆弱性修正方法の選択）
- FR-2で承認されたカテゴリ外の新規ライブラリ導入

### 新規ライブラリのカテゴリ事前承認（FR-2で確認）

FR-2技術選定の承認時に以下のカテゴリを承認済みとして扱う：

| カテゴリ | 例 | 自動実行 |
|---|---|---|
| テスト用ライブラリ | jest, pytest, vitest | 可 |
| 型定義 | @types/* | 可 |
| 開発用ユーティリティ | eslint plugins, prettier | 可 |
| ランタイム依存 | express, fastapi, prisma | FR-2で個別承認が必要 |

ランタイム依存は実装中に新たに必要性が判明した場合、Orchestratorが一旦停止してユーザーに確認する（作ってから修正しない原則）。

## 9. コードコメントの要件ID記載粒度

### 記載単位：関数（メソッド）単位

```typescript
/**
 * JWTトークンを生成する
 * @req REQ-2.1
 */
export function generateToken(payload: TokenPayload): string { ... }

/**
 * JWTトークンを検証する
 * @req REQ-2.2, REQ-2.3
 */
export function verifyToken(token: string): TokenPayload { ... }
```

複数の要件を担う関数はカンマ区切りで複数IDを記載する。

### 汎用コードの扱い

特定の要件に紐付かないユーティリティ・共通処理には要件IDを付与しない。

```typescript
/**
 * 入力値のサニタイズを行う共通ユーティリティ
 * @util
 */
```

`@util` / `@shared` のコードはトレーサビリティマトリクスに含めない。FR-7コード品質軸でのみ評価する。
