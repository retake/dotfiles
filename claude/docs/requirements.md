# Claude Code 自律エージェント活用 要件定義

## 業務要件

### 目的

やりたいことを入力したら、サブエージェントが自律的に設計・実装・レビューを完遂し、ユーザーは結果を確認するだけで済む開発フローを確立する。

---

### スコープ

| 対象 | 内容 |
|---|---|
| IN | 設計・実装・テスト・レビューの自律実行 |
| IN | 要件→実装のトレーサビリティ確保 |
| OUT | push・デプロイ（人間が実行） |
| OUT | 他AIとの連携・ファイル/コマンド起動（将来検討） |
| OUT | 実装中のリアルタイム要件変更検知（将来検討） |

---

### エージェントの自律境界

| 判断 | 方針 |
|---|---|
| 要件が曖昧（入力テンプレートの必須項目が未記載） | 着手前に人間へ確認・停止 |
| 技術選定 | 着手前に人間へ確認・停止 |
| スキル実行 | 自動（確認不要） |
| 設計 | 設計サマリを出力し、人間の承認後に実装へ進む |
| テスト失敗 | 3回まで自律対応、以降エスカレーション |
| セキュリティ判断 | 自動実行禁止・人間へ確認 |
| push・デプロイ | 人間が実行 |

※ 要件の変更検知は行わない。完了サマリで人間が要件との乖離を確認する。
※ 技術選定が入力テンプレートに明示されている場合は確認をスキップして進む（要件・完了条件の確認はスキップ不可）。

---

### 要件IDの規則

- `requirements.md` の見出しに明示的なIDを付与する（例：`## REQ-2.3 ログイン機能`）
- 自動採番は使わない（ツールによる振り直しを防ぐため）
- セクションを修正しても番号は変えない（内容変更のみ）
- セクションを削除した場合は番号を欠番にする（再利用しない）
- 新規セクションは末尾に追加する（既存番号の間に挿入しない）

---

### 完了の定義（エージェントの成果物）

1. 実装コード（テスト・lint通過済み）
2. コードコメント（対応する要件ID・実現方法を記載）
3. トレーサビリティマトリクス（要件ID→実装箇所→検証方法）
4. ターミナルへの完了サマリ（変更内容・要件との乖離の有無・人間が確認すべき点）

---

### タスク入力テンプレート

```markdown
## 目的
<!-- 何を解決するか。1〜3文で。 -->

## 完了条件
<!-- 検証可能な形で記述する。「〜できること」「〜が返ること」など。 -->
-

## スコープ
### 対象
-

### 対象外
-

## 制約
<!-- 技術・環境・パフォーマンス・セキュリティ上の制約があれば記載。 -->
- （なければ省略可）

## タスク規模
<!-- small（上限50回）/ medium（上限150回・デフォルト）/ large（上限300回） -->
- （省略可）

## テスト戦略
<!-- 省略時のデフォルト：単体テストのみ -->
- （省略可）

## 備考
<!-- 参照すべきドキュメント・既存コード・関連要件IDなど。 -->
- （省略可）
```

**曖昧と判定する条件（エージェントが確認停止するトリガー）：**

- 「目的」または「完了条件」が未記載
- 完了条件が検証不可能な形（「うまく動くこと」等）
- スコープが他モジュール・他タスクと重複する可能性がある
- 制約にセキュリティ要件が含まれるが具体的でない

---

## 機能要件

### FR-1. 入力受付・解釈

- 入力テンプレートを受け取り、必須項目（目的・完了条件）の有無を確認する
- 必須項目が未記載、または完了条件が検証不可能な場合は確認停止する（スキップ不可）
- 技術選定が明示されている場合はFR-2をスキップして進む

### FR-2. 技術選定

- 技術が未指定の場合、選択肢・トレードオフ・推奨理由を提示して確認待ちする
- 入力に技術が明示されている場合はスキップして進む

### FR-3. 設計

- モジュール構成・インタフェース定義を生成する
- 設計サマリをターミナルに出力し、人間の承認を待つ（承認後にFR-4へ進む）
- 設計内容を成果物として出力する（後工程のトレーサビリティ起点）

### FR-4. 実装

- 設計に基づいてコードを生成する
- 各コードに対応する要件ID（`REQ-x.x`）と実現方法をコメントとして記載する

### FR-5. テスト実装・実行

- 完了条件に基づいてテストコードを生成する
- テストを実行し、失敗した場合はタスクサイズとエラー種別に応じた上限回数まで自律修正する
  - 通常エラー（型エラー・ロジックエラー等）: small=2回 / medium=3回 / large=5回
  - 外部依存エラー（ネットワーク・タイムアウト等）: サイズ問わず1回
- 上限を超えても解消しない場合は確認停止する（停止箇所・失敗理由・試行内容を記録）

### FR-6. lint

- プロジェクトのlint設定（CLAUDE.md参照）に基づいてlintを実行する
- 指摘はタスクサイズとエラー種別に応じた上限回数まで自律修正する
  - 通常エラー（lint指摘）: small=2回 / medium=3回 / large=5回
  - 外部依存エラー: サイズ問わず1回
- 上限を超えても解消しない場合は確認停止する（停止箇所・失敗理由・試行内容を記録）

### FR-7. レビュー

以下の4軸で確認する：

| 軸 | 確認内容 |
|---|---|
| 要件整合 | 完了条件をすべて満たしているか |
| トレーサビリティ | 全要件IDに対応する実装・検証方法が存在するか |
| コード品質 | 設計の妥当性・可読性・重複の有無 |
| セキュリティ | OWASP Top10等の観点で問題がないか |

- セキュリティ上の問題を検出した場合は確認停止する（自動修正禁止）
- コード品質のロジック上の不具合（nullチェック漏れ・境界値エラー等）は自動修正してよい
- 自動修正を行った場合はFR-5・FR-6を再実行して「テスト・lint通過済み」を維持する
- インタフェース変更が必要な場合は確認停止する（FR-4からやり直し、上限1回）
  - やり直し前にdesign-summary.mdの更新を人間に促す確認停止を挟む
  - FR-5・FR-6の自律修正試行回数はリセットする

### FR-8. 成果物出力

タスク完了時に以下を出力する：

| 成果物 | 内容 |
|---|---|
| 実装コード | テスト・lint通過済み |
| コードコメント | 要件ID・実現方法を記載 |
| トレーサビリティマトリクス | 要件ID→実装箇所→検証方法 |
| 完了サマリ（ターミナル） | 変更内容・要件との乖離の有無・確認が必要な点 |

### FR-9. 確認停止・通知

- 停止時はターミナルに以下を出力する：停止箇所・理由・人間が取りうる選択肢
- 別途通知（将来検討）

### FR-10. エスカレーション記録・再開


- 確認停止・テスト/lint失敗時に以下をファイルに記録する：停止箇所・失敗理由・試行済みの対策・再開ポイント
- 記録先：`.claude/task-state.md`（プロジェクトルート直下）
- 再開トリガー：手動（人間が対応後にエージェントを再起動する）
- 再開時はエージェントが `task-state.md` を読み込み、記録された再開ポイントから処理を継続する

## 非機能要件

### NFR-1. セキュリティ

- エージェントの読み書きはプロジェクトディレクトリ内に限定する
- 以下には触れない：`~/.ssh`・認証情報ファイル・インフラ設定
- ネットワーク接続は許可する（パッケージ取得等のため）
- セキュリティ上の判断が必要な場合は自動実行せず、人間へ確認・停止する（FR-7参照）

### NFR-2. 観測可能性

エージェントの実行内容をツール呼び出し単位で記録する。

| 記録対象 | 内容 |
|---|---|
| 各ツール呼び出し | 操作内容・結果 |
| 各FR完了時 | 処理サマリ |
| 確認停止・エスカレーション | イベント・理由・選択肢 |

- 記録先：`.claude/agent-log.md`（プロジェクトルート直下、task-state.mdとは別ファイル）

### NFR-3. パフォーマンス（実行上限）

タスク規模を入力テンプレートで指定し、上限をツール呼び出し回数で管理する。

| タスク規模 | 上限回数 | 例 |
|---|---|---|
| small | 50回 | 関数1つの追加・バグ修正 |
| medium | 150回 | 機能1つの追加 |
| large | 300回 | 複数モジュールにまたがる実装 |

- 省略時のデフォルト：medium（150回）
- 各FRのチェックポイントで累計呼び出し回数を `.claude/task-state.md` に記録する
- 上限超過時は確認停止し、`.claude/task-state.md` に記録する
