# 開発スタイル

## 基本方針

- ソロ開発。目的は特定プロダクトの完成より**再現性ある開発スタイルの確立**

## 要件定義

> **要件の内容（正本）は各 repo の `docs/product-request.md` / `docs/requirements.md` / `AGENT_GUIDE.md` を参照する。この節は「要件をどう扱うか」の原則（meta-process）であり、要件そのものは repo docs に委譲する。**

- 目的・スコープ・制約・判断基準を最初に明確化する
- 曖昧な要件はそのまま進めず、論点を整理して確認する
- 機能一覧より「何を解決するか」「何を解決しないか」を先に定義する

## 実装

- 技術選定時は選択肢・トレードオフ・推奨理由を明示する
- スモールスタート：動く最小構成から始め、段階的に拡張する
- 選択肢のある実装（ツール・バージョン・設定値等）は作る前に確認する。作ってから修正しない
- 環境依存の操作（バイナリ取得・インストール等）の前にOS・アーキテクチャ等の前提を確認する。普段使う環境のカタログは `~/dev/environments/_index.md` にあるので、ライブラリ追加・ビルドツール選定の前にまず参照する
- ネイティブ依存ライブラリ（FFI / C++ プラグイン等）を追加する前に、bundled binary のアーキテクチャ・compile-time マクロ・対象 OS を確認する。事後の linker error / runtime 無音は手戻りが大きい
- pub-cache / node_modules 等の管理外ツリーを直接編集するのは「動作確認のための一時手段」に限る。永続化策（fork / postinstall script / pubspec override）を決めずに依存しない
- 既存設定の移植時は「なぜあるか・今も使うか」を確認してから判断する。あるから持ち込まない
- 外部リソース（ディレクトリ・ファイル・リポジトリ）の存在・状態に言及する前に、必ず実在を確認する
- リネーム／用語統一リファクタは新規テストを書かない。既存テストの緑維持が回帰条件
- 用語揺れを解消する際は「名前の根拠がある方（enum/class 名などの第一定義）」を正とし、派生・コメント・テスト文言を寄せる
- 巨大ファイル分割リファクタでは、元ファイルに `export ... show` を残す re-export パターンで呼び出し元（他ファイル + テスト）を無改修で通す。分割 PR を小さく保てるうえ、移行期の import 一括差し替えコストが消える。分割は Step 単位で `format` / `lint` / `test` を各段階で回し、「どのコミットで壊れたか」を即特定できる構造を維持する

## テスト

- 実装と並走してテストを設計する（後付けしない）
- テスト戦略（単体・結合・E2Eの範囲）を実装前に合意する
- ハーネスのセットアップをCIに組み込む（手動実行に依存しない）
- 改修規模の見積もりは行数ではなく、既存の Fake 注入点・抽象インタフェース・純関数化の有無で決まる。大規模に見える改修でもハーネスが揃っていれば TDD で想定より短時間で収束する
- バグ修正のテストは「修正を一時巻き戻して赤転するか」まで確認してから fix を再適用する。緑のみの確認だと本症状の再現になっておらず、機能保証テストに留まる（例: SnackBar Undo の ref dispose 問題は、単純な Undo タップでは再現せず、StandbyView を unmount させる専用ハーネスが必要だった）

ハーネス設計の問いかけ（技術スタックを問わず共通）：
- **何を封じるか** — 副作用・外部依存（ファイルシステム・ネットワーク・環境変数）を制御する
- **何を注入するか** — 入力・状態・コンテキストをテスト側から与える
- **何を観測するか** — 出力・状態変化・副作用の痕跡を検証する

### 状態を持つ UI コンポーネントのルール

`StatefulWidget` や「複数の表示モード・ページ送り・チェック状態」を持つ UI コンポーネントを追加・変更するときは、以下を実装前に行う：

1. **状態一覧を列挙する** — コンポーネントが取りうる状態と、それぞれで何が表示されるかをコメントまたはテストケース名として書く。例：「3件以下→全件表示・矢印なし」「全部チェック→完了メッセージ」
2. **境界条件をテストケースに落とす** — 「N件以下」「N件以上」「0件」「全チェック済み」などの境界をテストとして書いてから実装する
3. **ウィジェットテストは専用ファイルに分離する** — シナリオテストに混ぜず `widgets/*_test.dart` または `presentation/*_display_test.dart` として独立させる

これにより「実装→実機で触って発見→fix→また発見→fix」のサイクルを防ぐ。

### UI インタラクションチェックリスト

UI コンポーネントを追加・変更するときは、以下を実装前に確認する：

1. **取り消し不可操作の安全性** — 削除・上書き・送信など元に戻せない操作には、SnackBar Undo・確認ダイアログ・キャンセルボタンのいずれかを必ず用意する
2. **制約の可視化** — ボタンを無効にする場合は「なぜ無効か」を近くに表示する。ユーザーが理由を推測する UI にしない
3. **同一見た目の動作一貫性** — 同じ見た目の要素が状態によって異なる動作をする場合、それを意図的な設計として明示する。暗黙の分岐は混乱を生む
4. **タッチターゲットサイズ** — インタラクティブな要素は最小 44dp を確保する。特にモバイル Web を対象とする場合は必ず確認する
5. **設計方針との照合** — プロジェクトに明示された設計方針（ADHD配慮・情報最小限・色で緊急度を伝える等）を requirements.md で確認し、各 UI 決定がその方針と整合しているかをチェックする
6. **レイアウトシフト防止** — 状態変化で Column に子要素を追加/削除すると、`Spacer` を含む列では他の要素（時刻表示等）の位置がずれる。対策: `AnimatedSwitcher` でコンテナ**内の**既存要素を置き換え、外側の高さを変えない設計にする。「完了メッセージ」のような新状態表示は別ウィジェットとして追加するのではなく、既存要素（ヘッダーテキスト・矢印ボタン）とのスワップとして実装する
7. **ボタン文言は機能語ではなく状態表明に寄せる** — 「+3分延長」「スキップ」「完了」のような操作名（機能語）より、「まだ出発しない」「今回の準備を停止」のような「ユーザーが今何をしているかの状態表明」を優先すると、ADHD 配慮だけでなく朝・焦り・緊急時の判断コストが下がる。特に救済系・中断系ボタンでは自己状態を言語化する文言が受容性を高める。ただし場所を限定する表現（「まだ家にいる」等）は状況を仮定しすぎるため避け、中立な状態語を選ぶ

### プライベートメソッドの配置ルール

実装前に3問を問う：

1. **「何が正しいか・いつ実行するか・どう変換するか」を決める計算か？**
   → YES → `domain/` の適切なクラスのパブリックメソッドに配置する

2. **複数のリポジトリ・サービスを協調させるユースケースか？**
   → YES → `application/` のユースケースクラスまたは Notifier のパブリックメソッドに配置する

3. **「この画面の構造を組み立てる」純粋な UI 補助か？**
   → YES → `_build*()` としてプライベートで可

プライベートメソッドで許容されるのは 3 のみ。1・2 に該当するものをプライベートに書いた場合は、lint 違反と同様に「流れの中で」適切な層に移動してからコミットする。

- 実装着手前に `~/dev/ai-coding-principles.md` の AI 共通原則を確認する

## Git運用

- 運用ルール（コミットメッセージ・ブランチ戦略）は初回プロジェクト時に合意して以降統一する

## Lint・CI/CD

- lintは実装開始前に設定する（後付けしない）
- 設定ファイルはリポジトリにコミットする（ローカル依存・暗黙の設定にしない）

## 成果物管理

- 各プロジェクト：`CLAUDE.md` と `docs/`（product-request.md → requirements.md・test-plan.md）を置く。`product-request.md` が要求層の最上流正本。**要件の内容・層定義はすべて repo docs が source of truth。この文書に要件の内容を重複記載しない**
- 振り返り：`~/retrospectives/_index.md`（dotfiles管理）、個別ファイルはローカルのみ
- 運用原則（episodic 教訓を抽象化したもの）：`~/retrospectives/_principles.md`。新しい教訓は `_index.md` 未分類 episodic に追加し、同型が3件以上溜まったら `/retro` が原則への昇格を提案する
- 開発環境カタログ：`~/dev/environments/_index.md`（普段使う環境ごとに 1 ファイル。OS・arch・既知制約・ライブラリ落とし穴を記録）
- プロジェクト作業時はそのリポジトリの `CLAUDE.md` と `docs/` を参照する

# Codex連携

Claude Code主担当、Codex補助の体制。共通ルールは各リポジトリの `AGENT_GUIDE.md` に集約する。依頼は `~/.claude/docs/codex-request-template.md` に沿う。

## handoff loop

継続対話が必要で、repo に `AGENT_GUIDE.md` と `docs/agent-handoff-template.md` がある場合は
`claude-codex-handoff-loop.sh` を使って Codex との bounded handoff loop を開始または継続できる。

原則:

- 相談ループ (`Type: design-consult`) は `done + Human` で止める。ただし Findings がコスト極小（文言変更・1ファイル修正程度）で Human 承認待ちのコストが不釣り合いな場合は Claude Code が直接実装してよい
- 修正ループ (`Type: implementation / review / bug-triage / audit / security-audit`) は reviewer 確認後 `archived` を基本にする
- Codex は handoff ファイルのみ更新し、実装ファイルは Claude Code が更新する

開始方法:

- slash command が使える場合:
  `/handoff-loop docs/agent-handoff-claudecode-foo-YYYYMMDD.md`
  `/handoff-loop HO-001`
- 直接実行する場合:
  `claude-codex-handoff-loop.sh --repo "$(pwd)" --handoff <path>`
  `claude-codex-handoff-loop.sh --repo "$(pwd)" --task-id HO-001`

## Codex review はリスクベース

以下に該当する場合のみ Codex review を実施する（小規模改修・typo・文言変更・局所 UI 修正は不要）:

- auth/session/token/crypto 関連の変更
- Timer/Stream/dispose 等の非同期ライフサイクル変更
- 大きなリファクタリング後
- ユーザーが不安を表明した場合
- orchestrate FR-10完了後にリスクレベルが high の場合
