# Retro — タスク完了後処理スキル

orchestrate完了後（または手動タスク完了後）に実行する後処理スキル。
draft が存在しない／古い場合はセッション内容から対話的に振り返りを起こし、
恒久ファイルへ昇格させて教訓を蓄積する。

## 実行手順

### ステップ0: draft の新鮮度判定と対話的起票（手動タスク向け）

目的: orchestrate を通らない手動作業でも、`/retro` だけで振り返りを完結させる。

1. **draft の状態を分類する**：
   - **不在**: `.claude/retrospective-draft.md` が存在しない
   - **古い**: draft が既に `~/retrospectives/` に同一内容でコピー済（`_index.md` の振り返り一覧に登場するタイトル・タスクIDと一致する）、かつ最終更新より後に新規コミットがある
     - 判定例: `git log --since="$(stat -c %y .claude/retrospective-draft.md)" --oneline` が非空
   - **有効**: 上記いずれにも該当しない（直近タスクの draft）

2. **分類に応じて分岐**：
   - **有効** または 引数に `--auto` が含まれる場合: 対話をスキップしてステップ1へ
   - **不在／古い** かつ `--auto` 指定なし: 以下の対話型起票に進む

3. **対話型起票の手順**：
   1. セッションの素材をユーザーに表示する:
      - `git log --oneline -10`（直近コミット）
      - `git diff --stat HEAD~{N}..HEAD`（N は「今回セッションで作られたコミット数」。古い draft 以降のコミットが目安）
      - `git status --short`（untracked 含む未コミット差分）
   2. ユーザーに 4 問を一括提示する（回答は 1 メッセージでまとめて書いてもらう）:
      - **Q1: 実装サマリ** — 何を実装したか（対象 REQ／主要ファイル／テスト件数）
      - **Q2: 発生した問題** — 詰まった点・手戻り・想定外の挙動（なければ「なし」）
      - **Q3: 教訓候補** — 次回も再現可能な学び（シングルループ：手順改善／ダブルループ：前提の問い直し）
      - **Q4: アクションアイテム** — 残作業・判断待ち・追加検討項目
   3. ユーザー回答をもとに下記フォーマットで `.claude/retrospective-draft.md` を上書きする:
      ```markdown
      # 振り返り: {プロジェクト名} {タスク概要} {YYYY-MM-DD}

      タスクID: {TASK-YYYYMMDD-NNN または 手動タスクなら「手動実装・orchestrate非経由」と明記}
      完了日時: {YYYY-MM-DD HH:MM}

      ## 実装サマリ
      （Q1 の回答を整形）

      ## 発生した問題とエスカレーション
      （Q2 の回答を整形。なければ「特になし」）

      ## 教訓候補（シングルループ）
      （Q3 のうち手順改善系）

      ## 前提の問い直し（ダブルループ）
      （Q3 のうち前提見直し系。なければ省略）

      ## アクションアイテム
      （Q4 を `- [ ]` チェックボックス形式で列挙）

      ## _index.mdへの追加候補
      （教訓候補から、他プロジェクトでも再現可能な汎用性あるものだけを抽出。
       末尾に `（YYYY-MM-project）` を付ける）
      ```
   4. draft を書き出したら「draft を更新しました。続けて処理しますか？」と確認し、承認後にステップ1へ進む

4. **対話スキップ条件の追加**: 直前の会話ですでに同等の情報が揃っている（ユーザーが /retro 直前に作業サマリを文章で書いた等）場合は、4 問提示を省略して draft を直接起こしてよい。ただし「この内容で draft を起こします」と一度表示して承認を取ること

### ステップ1: ドラフトの確認

1. プロジェクトの `.claude/retrospective-draft.md` を読む
   - 存在しない場合（ステップ0で起票されなかった異常系）: 「振り返りドラフトが見つかりません」と表示して終了する
2. ドラフトから以下を抽出する：
   - **タイトル行**（`# 振り返り: …`）からプロジェクト名・日付を取得
   - **タスクID・完了日時**（メタ情報行）
   - **`_index.mdへの追加候補`** セクションの箇条書き（教訓として追加する）
   - **`アクションアイテム`** セクションの `- [ ]` 未完了項目（自動適用できるものを仕分ける）

### ステップ2: 保存先ファイル名の決定

引数またはタイトルから保存先ファイル名を決定する：

- ファイル名フォーマット: `YYYY-MM-project.md`（例: `2026-04-alarm.md`）
- 引数が指定された場合: `YYYY-MM-{引数}.md` で採番（`date +%Y-%m` で年月取得）
- 引数が省略された場合: タイトル行から推定（推定結果をユーザーに表示してから続行）
- すでに同名ファイルが `~/retrospectives/` に存在する場合: 上書きせず `YYYY-MM-{プロジェクト名}-2.md` で採番する

### ステップ3: ドラフトをコピー

```bash
cp .claude/retrospective-draft.md ~/retrospectives/{ファイル名}
```

コピー後、コピー元（`.claude/retrospective-draft.md`）は削除しない（orchestrate が参照するため）。

### ステップ4: `~/retrospectives/_index.md` の更新

`~/retrospectives/_index.md` を読んで以下を更新する：

**4-1. 教訓サマリへの追記**

`_index.mdへの追加候補` セクションの各箇条書きを「教訓サマリ」に追加する。
- すでに同一または類似の教訓が存在する場合はスキップ（重複確認）
- 追加する箇所: 既存の最後の教訓行の直後

**4-2. 振り返り一覧への追記**

「振り返り一覧」テーブルに新しい行を追加する：

```
| {完了日時の日付部分} | {プロジェクト概要} | [{ファイル名}]({ファイル名}) |
```

- プロジェクト概要はドラフトのタイトル行から抽出する
- すでに同名ファイルへのリンクがある場合はスキップする

### ステップ5: ドキュメント修正の適用

`docs/completion-summary.md` と `docs/design-summary.md` に実装との乖離が記録されている場合、自動適用できるものを適用する。

**対象とする乖離（自動適用可）：**
- `completion-summary.md` の「人間が確認すべき点」に「design-summary.mdの〜を修正」のような指示がある
- `design-summary.md` 内に実装と異なる記述が特定されている（コントラスト閾値・色・API名など）

**適用手順：**
1. `docs/completion-summary.md` の「人間が確認すべき点」を読む（存在する場合）
2. 「design-summary.md の〜」形式の修正指示を抽出する
3. `docs/design-summary.md` 該当箇所をgrepで特定し、Editで修正する
4. 修正した箇所をレポートに含める

**自動適用しない乖離（手動対応が必要なもの）：**
- UIの実装変更を伴うもの（コード修正が必要）
- 判断を要する設計変更

### ステップ6: アクションアイテムの仕分けと報告

ドラフトの `アクションアイテム` を「自動適用済み」「手動対応が必要」に仕分けて表示する。

```
[RETRO] 後処理完了

## 保存先
~/retrospectives/{ファイル名}

## _index.md への追加
教訓 {N}件 追加:
  - （追加した教訓1）
  - （追加した教訓2）
  ...
一覧: 「{プロジェクト概要}」を追加

## ドキュメント修正
（適用した修正の一覧、または「なし」）

## 手動対応が残っているアクションアイテム
- [ ] （未対応のアクション1）
- [ ] （未対応のアクション2）
（なければ「なし」）
```

### ステップ7: 追加 persist の確認

目的: retro（＝振り返りファイル + `_index.md` の教訓）では拾いきれない、
**セッション外に持ち越したいルール・設定・嗜好** を明示的に問う。「なし」が
普通の答えで、あってもどの層に保存するかをユーザーに選んでもらう。

1. ユーザーに一度だけ問う:

   ```
   ## 追加 persist の確認
   
   今日のやり取りで、`/retro` の教訓以外に以下のいずれかに保存したいものは？
   
   - memory（auto-memory）: ユーザー嗜好・作業癖・フィードバック
   - ~/.claude/CLAUDE.md（root）: 応答スタイル・実行判断ルール
   - ~/dev/CLAUDE.md（dev / 言語 FW 非依存の方法論）: 開発慣習・AI 運用術
   - {project}/CLAUDE.md: プロダクト固有の落とし穴・慣用パターン
   - ~/.claude/settings.json: hook・allow リスト・ハーネス設定
   
   「なし」でも問題ありません。
   ```

2. ユーザーが提示した項目ごとに層を判定し、具体の追記案を提示する（判定基準は下表）:

   | 保存したい内容 | 推奨層 |
   |---|---|
   | 本人の好みや継続的な要望（例: コミットメッセージは日本語） | memory |
   | 応答トーン・判断ルール（例: 迷ったら即決を求める） | root CLAUDE.md |
   | 言語 FW を問わない開発方針（例: リネーム系は新規テスト不要） | dev CLAUDE.md |
   | 特定 FW・言語の落とし穴（例: ConsumerStatefulWidget dispose で ref 無効） | project CLAUDE.md |
   | ツール・hook・自動許可（例: `Bash(flutter test*)` の allow 追加） | settings.json |

3. ユーザー承認を得た項目のみ適用する:
   - memory: auto-memory の仕組みで書き出す（`~/dotfiles/claude/memory/global/` 配下）
   - CLAUDE.md: Edit で既存節へ追記（新規節を作る場合も最小限）
   - settings.json: Edit で既存構造を壊さずキーを追加

4. すべての persist が終わったら「他にありますか？」と再確認し、「ない」で終了する
   （適用した項目は最終サマリに `## 追加 persist` として列挙する）

5. **スキップ条件**: 以下のいずれかに該当する場合、ステップ 7 の問いかけを省略してよい:
   - `--auto` 引数が指定されている
   - 直前のセッション内で既に persist 系の作業を完了している（例: /retro 直前に CLAUDE.md や memory へ自発的に追記済み）

## 制約

- `.claude/retrospective-draft.md` は削除しない
- `~/retrospectives/` 以外の書き込みは以下に限る:
  - ドラフト乖離の適用（ステップ5）: `docs/design-summary.md` のみ
  - 追加 persist（ステップ7）: ユーザー承認後の CLAUDE.md / settings.json / memory のみ
- コード・テストの修正は行わない
- git commit / push は行わない（追記したファイルは `git status` で見せて終わる）
- ステップ 7 でユーザーが「なし」と答えたら、それ以上掘らない（押し付けない）
