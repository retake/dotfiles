# Audit Handoffs — ハンドオフ指摘の全件確認スキル

`docs/agent-handoff-*.md`（Codex/別エージェントからのレビュー・調査文書）に記載された指摘を全件抽出し、現状コードと照合して対応状況を判定・対応案を提案する。

## 使い方

- `/audit-handoffs` — `docs/agent-handoff-*.md` を対象（通常フロー）
- `/audit-handoffs <glob>` — 引数で対象パターンを上書き
  - 過去対応済みも含めて確認: `/audit-handoffs 'docs/**/agent-handoff-*.md'`
  - archive のみ: `/audit-handoffs docs/archive/*.md`

## アーカイブ運用

対応完了したハンドオフは `docs/archive/` 配下に `git mv` で移動する。
通常フロー（引数なし実行）ではアーカイブ対象外となり、未対応・要判断
のものだけが監査対象になる。

## 実行手順

### ステップ1: ハンドオフファイルの列挙

`ls docs/agent-handoff-*.md 2>/dev/null` を1回実行する。untracked も含めてファイルシステム実体を直接取得できるため、これで十分。

件数を表示してユーザーに確認を求める（0件なら終了）。

### ステップ2: 各ファイルの抽出

**基本は Grep で必要箇所のみ取得する**。`Read` でファイル全体を読まない。

各ファイルに対して以下の順で実行する：

1. `Grep` でタイトル行（`# Agent Handoff`）と Type 行（`## Type`）を取得
2. `Grep` で `## Findings`・`### [0-9]`・`severity`・`該当箇所`・`提案:` を含む行を取得
3. 書式が独特で Grep 結果が不十分な場合のみ `Read` にフォールバックする

読み取る内容：
- タイトルと出所推定（Codex / Claude / 他）
- Type（review / bug-triage / design-consult / audit / security-audit 等）
- 各指摘の severity（High / Medium / Low）または優先度
- 各指摘の該当ファイル・行番号と提案内容

### ステップ3: 現状コードとの照合（Explore サブエージェントに委譲）

**トークン節約の原則**: 生の Grep 結果・git log・Read 出力をメイン文脈に蓄積しない。照合作業はすべてサブエージェントに委譲し、結果テーブルだけ受け取る。

ステップ2で抽出した指摘一覧を使い、**Explore サブエージェント**を起動してコード照合を委譲する。メイン文脈に生の Grep 結果や git log を蓄積しない。

**サブエージェントへのプロンプト構成（自己完結で渡すこと）：**

```
以下の指摘一覧について、現状コードと git log を照合し、各指摘の対応状況を返してください。
Read でファイル全体を読まず、Grep・git log・局所 Read（±10行）で判定してください。

[指摘一覧]
- 指摘1: ファイル=xxx.dart, キーワード=yyy, 内容=...
- 指摘2: ...

[照合手順]
1. git log --oneline -60 を1回取得（全指摘で共有）
2. docs/traceability.md があれば該当要件IDのみ Grep（1回で全指摘分まとめて）
3. 全指摘の Grep を1メッセージで並列実行する（指摘ごとに逐次実行しない）
4. Grep 結果が曖昧な指摘のみ、局所 Read（±10行）で補完

[返答フォーマット]
| # | キーワード | Grep結果 | 関連コミット | 暫定判定(✅/❌/⚠️/⚪/❓) | 根拠 |
```

サブエージェントの返答（照合結果テーブル）を受け取ったらステップ4に進む。

### ステップ4: 対応状況の分類

各指摘を以下に分類する：

| 区分 | 条件 |
|---|---|
| ✅ 対応済 | コードが指摘を解決している + 対応コミットが存在する |
| ⚠️ 部分対応 | 指摘の一部のみ解決、または traceability 記載ありだが実装が完全でない |
| ❌ 未対応 | 指摘の問題パターンが現存する |
| ⚪ 要判断 | 実装判断が要件レベル（採用/不採用の意思決定待ち） |
| ❓ 判定不能 | 照合不能（外部システム依存・手動検証など） |

判定に迷う場合は ⚠️ または ❓ に寄せ、理由を明記する。

### ステップ5: サマリと対応案の出力

以下のフォーマットで出力する：

```

- ...
```

## ステップ6: 対応案の提案（c 範囲）

残課題それぞれについて以下を必ず含める：

- **推定コスト**（小/中/大）
- **対応案**: 具体的なコード変更の方向性（ファイル・関数・パッチの概要）
- **優先度の根拠**: なぜ高/中/低なのか（影響範囲・セキュリティ・UX）

対応案は「実装する」ではなく「提案する」に留める。実装は別途ユーザーの承認を得てから。

## 制約

- 既存のコードやドキュメントを編集しない（照合とレポートのみ）
- git push・git commit は行わない
- 判定に自信が持てない場合は ⚠️ や ❓ に寄せて根拠を明記する
- ハンドオフの書式が独特でパースできない場合は、元ファイルを Read して人手で読む
- untracked の handoff ファイルも必ず含める（Claude Code のセッション中に後から追加されるケースを想定）

## 過去の反省に基づく補足

- 以前、`ls docs/` の初回結果をキャッシュして判断したため、セッション途中で追加された handoff 2件を見落とした事故があった。**毎回ステップ1の列挙を実行する**（キャッシュに頼らない）
- 「対応済み」と書かれていても、コードとの照合を省略しない（ドキュメントの記述と実装がズレている可能性がある）
