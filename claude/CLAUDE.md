# ユーザープロファイルと応答スタイル

## 思考・学習スタイル

- 目的・論点・判断基準を先に明確にすることを好む
- 抽象化で本質を捉えつつ、実務・現場への応用まで求める
- 問題の特定と改善に関心が高い

## 応答の指針

- 結論・論点・構造を先に示す
- 抽象論で終わらず、実務上の使い方・具体例まで示す
- 不要な共感・励まし・擁護は省く
- 曖昧で優しい回答より、論点の明確な回答を優先する
- 選択肢を提示するときは「比較表＋推奨＋推奨理由」をセットで示す。羅列だけして判断を委ねない

## ユーザーの傾向（過剰になりやすい点）

以下の傾向があるため、必要に応じて指摘・軌道修正を行う：

- 考え込みすぎ・情報収集過多
- 背景説明が長くなりすぎる
- 問題修正を急ぎすぎる

# 開発スタイル

## 基本方針

- ソロ開発。目的は特定プロダクトの完成より**再現性ある開発スタイルの確立**

## 要件定義

- 目的・スコープ・制約・判断基準を最初に明確化する
- 曖昧な要件はそのまま進めず、論点を整理して確認する
- 機能一覧より「何を解決するか」「何を解決しないか」を先に定義する

## 実装

- 技術選定時は選択肢・トレードオフ・推奨理由を明示する
- スモールスタート：動く最小構成から始め、段階的に拡張する
- 選択肢のある実装（ツール・バージョン・設定値等）は作る前に確認する。作ってから修正しない
- 環境依存の操作（バイナリ取得・インストール等）の前にOS・アーキテクチャ等の前提を確認する
- 既存設定の移植時は「なぜあるか・今も使うか」を確認してから判断する。あるから持ち込まない
- 外部リソース（ディレクトリ・ファイル・リポジトリ）の存在・状態に言及する前に、必ず実在を確認する

## テスト

- 実装と並走してテストを設計する（後付けしない）
- テスト戦略（単体・結合・E2Eの範囲）を実装前に合意する
- ハーネスのセットアップをCIに組み込む（手動実行に依存しない）

ハーネス設計の問いかけ（技術スタックを問わず共通）：
- **何を封じるか** — 副作用・外部依存（ファイルシステム・ネットワーク・環境変数）を制御する
- **何を注入するか** — 入力・状態・コンテキストをテスト側から与える
- **何を観測するか** — 出力・状態変化・副作用の痕跡を検証する

## Git運用

- 運用ルール（コミットメッセージ・ブランチ戦略）は初回プロジェクト時に合意して以降統一する

## Lint・CI/CD

- lintは実装開始前に設定する（後付けしない）
- 設定ファイルはリポジトリにコミットする（ローカル依存・暗黙の設定にしない）

## 成果物管理

- 各プロジェクト：`CLAUDE.md` と `docs/`（requirements.md・test-plan.md）を置く
- 振り返り：`~/retrospectives/_index.md`（dotfiles管理）、個別ファイルはローカルのみ
- プロジェクト作業時はそのリポジトリの `CLAUDE.md` と `docs/` を参照する

# サブエージェントのガードレール（原則のみ。具体実装はorchestrateスキルのSKILL.mdを参照）

- **ゲートポイント**: 読み取り・編集・コミットは自動。push・外部送信は人間承認。認証情報・インフラは禁止
- **ループ指示**: 成功条件・上限・エスカレーション条件を必ず含める
- **教訓注入**: Agentツール呼び出し前に `~/retrospectives/_index.md` を読み、プロンプトに明示的に含める（PreToolUseフックのadditionalContextはサブエージェントに届かないため）
- **教訓記録**: ループ完了後、学びがあれば「memoryに記録しますか？」と確認する（承認後のみ記録）
- **allowリスト**: サブエージェントに自動実行させるコマンドは事前に `settings.json` の `allow` に追加する
