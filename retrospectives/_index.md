# 振り返りインデックス

## 教訓サマリ

- 環境前提（OS・アーキテクチャ等）は操作の前に確認する（例: バイナリ取得前に `uname -m` でアーキを確認）（2026-03-dotfiles）
- 既存設定を移植するときは用途を確認してから持ち込む。あるから持ち込まない（例: dotfiles移植で不要設定が混入）（2026-03-dotfiles）
- 選択肢のある実装は作る前に確認する。作ってから修正しない（例: ツール・バージョン・設定値の選択は着手前に合意する）（2026-03-dotfiles）
- テストで呼び出すコマンドの外部依存（stdin等）はテスト側で封じてから検証する（例: stdin参照コマンドは `< /dev/null` で封じる。`[ -p /dev/stdin ]` はローカルとCIで挙動が異なる）（2026-03-dotfiles）
- バックグラウンドエージェントに自動実行させるコマンドは委譲前にallowリストへ追加しておく（例: Testerへpytestを委譲する前に `Bash(pytest*)` をsettings.jsonに追加する）（2026-03-dotfiles）
- 技術スタック確定時に依存ツールの存在確認とインストールを行う。実行フェーズ直前の発覚は手戻りになる（例: pip不在環境でのpytest・flake8はFR-2停止時にaptでインストールする）（2026-04-claude-set）
- フロー仕様の初稿は正常系だけでは不十分。「ファイル不在は？」「ESCALATEDは？」「カウントはbefore/after？」「DONEは1通りか？」の4問で例外系を埋めてからレビューに渡す（2026-04-claude-set）
- レビュアーループは上限5回でも構造的欠損（分岐がない）が残ると回数が嵩む。初稿で構造的欠損を潰せれば3回以内が目安（2026-04-claude-set）
