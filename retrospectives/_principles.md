# 運用原則（抽象化済み）

複数の具体教訓から抽象化した運用原則。個別の episodic（具体事例）は各振り返りファイルを参照。
新しい retrospective を追加した際は「既存原則の派生元に該当するか／新クラスタか」を判定し、該当すれば派生元に追記、3件以上の新しい同型が溜まれば新原則として追加する。

---

## P1: テストハーネスは「副作用・時間・位置」を封じる設計にする

テストで不安定化する主因は、副作用（外部コマンド・連続非同期・SnackBar キュー）・時間（Timer.periodic の実時間）・位置（find.byType の N 番目）を封じていないこと。
実装と同時にハーネス側で以下を用意する：

- **副作用**: stdin・外部プロセス依存はテスト側で `< /dev/null` 等で封じる。非同期ループ発火時は `pumpAndSettle` ではなく `pump()` + `pump(Duration)` で必要最小限だけ進める
- **時間**: Timer.periodic は `CountdownTimer` 相当の抽象を噛ませて tick を手動注入できる形にする
- **位置**: Widget finder は `find.byType(X).first` ではなく `find.widgetWithText(X, 'label')` でラベルで一意特定する
- **再現性**: バグ修正テストは「修正を一時巻き戻して赤転するか」で本症状の再現を確認する。緑のみは機能保証テストに留まる

改修規模の見積もりは行数ではなく「既存の純関数化・Fake 注入点・抽象インタフェースの有無」で決まる。ハーネスが揃っていれば大改修でも TDD で短時間に収束する。

## P2: 環境・プラットフォーム前提は着手前に確認する

事後の linker error / 無音 / 例外は手戻りが大きい。ライブラリ追加・ツール選定・実機体験の前に以下を確認する：

- **アーキテクチャ**: ネイティブ依存ライブラリの bundled binary arch・compile-time マクロ・対象 OS を確認する
- **ツール存在**: 技術スタック確定時に依存ツール（pytest・flake8 等）の存在確認とインストールまで行う
- **プラットフォーム分岐**: プラットフォーム依存パッケージ（wakelock_plus 等）は `kIsWeb` / `Platform` チェックで未対応環境を明示的に弾く。パッケージ側の例外に依存しない
- **実機検証の早期化**: 音声・グラフィックスなど環境依存度の高い体験は、選択肢の比較表を出す段階でアクションに「実機検証」を含める
- **管理外ツリーの扱い**: pub-cache / node_modules の直接編集は一時手段に限定し、永続化策（fork / postinstall / override）を決めてから依存する
- **dotfiles リンク**: シンボリンクはディレクトリ丸ごとではなく個別ファイルで張る

環境カタログ（`~/dev/environments/_index.md`）を着手前に参照する。

**派生元**: 2026-04-claude-set.md, 2026-04-alarm-4.md, 2026-04-alarm-6.md

## P3: サブエージェント運用は「境界・カウント基準・独立検証」で安定化する

サブエージェントは便利だが、委譲境界と停止条件を事前に決めないと迷走する。以下を徹底する：

- **allow リスト**: 自動実行させるコマンドは委譲前に settings.json の allow に追加する
- **呼び出し方法**: カスタムエージェント（`~/.claude/agents/`）は Agent tool の subagent_type ではなく `@agent名` で呼び出す
- **委譲の粒度**: 大規模コメント削除・import 整序のような機能に影響しない変更は、Reviewer に委ねず設計承認時に Implementer へ一括実施させる
- **レビューループのカウント基準**: 「ロジック不具合の修正のみ」を1周とカウントする。コメント削除・import 整序等は自動修正としてカウントしない。構造的欠損（分岐がない）があると5回でも足りない → 初稿で潰す
- **停止条件**: Tester が同一テストで3回連続失敗したら自律修正を続けず、手動検証移管か設計再考に切り替える
- **独立検証**: サブエージェントの「既存バグ / HEAD で再現」判定は、Orchestrator 側で `git stash` + 該当テスト単独実行で独立検証してから信じる

**派生元**: 2026-04-claude-set.md, 2026-04-alarm.md, 2026-04-alarm-2.md

## P4: ドキュメント同期は即時化し、用語は第一定義に寄せる

ドキュメントの遅延同期は traceability を崩す。以下の運用で乖離を防ぐ：

- **実装直後更新**: 手動実装でも実装直後に design-summary.md・traceability.md を更新する。orchestrate 非経由の修正でも同じリズムを適用する
- **2段運用**: `/audit-handoffs` → `/archive-handoffs` で残課題の視認性を維持する。採用完了したハンドオフは即 archive して /audit 対象から外す
- **用語の正規元**: 用語揺れは「名前の根拠がある方（enum / class 名などの第一定義）」を正とし、派生・コメント・テスト文言を寄せる。リネーム系リファクタは新規テスト不要、既存テストの緑維持が回帰条件

**派生元**: 2026-04-alarm.md, 2026-04-alarm-2.md, 2026-04-alarm-3.md, 2026-04-alarm-10.md, 2026-04-alarm-12.md

## P5: ユーザー報告は字面で受け取らず、切り分けと実ログで検証する

即「コード修正」に走ると誤った方向に時間を使う。以下の順で確認する：

1. **切り分け**: 現象が「コードの問題」「環境の問題」「ライブラリの問題」のどれかを最初に特定する
2. **実ログ優先**: 字面の症状ではなく、実ログ・スタックトレースを最優先で確認する
3. **再現性**: 再現手順を得てから修正に入る

**派生元**: 2026-04-alarm-6.md, 2026-04-alarm-7.md

## P6: Flutter/Riverpod の dispose・非同期は「参照の事前捕捉」で組む

dispose 後の ref アクセス・SnackBar キュー・一時的な状態変更の三領域で事故が起きやすい。パターンで固定化する：

- **dispose で ref を使わない**: `ConsumerStatefulWidget` の dispose では ref が無効。外部サービスへのクリーンアップは initState で参照をキャッシュ（`_service = ref.read(serviceProvider)`）して dispose で呼ぶ
- **delayed callback の参照捕捉**: SnackBar 等の delayed callback で WidgetRef を使わず、Notifier 参照をクロージャに事前捕捉する
- **SnackBar キュー**: ウィジェットテストの SnackBar 検証では、他所から出る先行 SnackBar がキューを占有していないか確認する
- **一時オーバーライドパターン**: 永続データを汚さず進行中セッションでのみ変更を効かせたい場合、state に `overrideX: T?` を追加し `effectiveX` getter で合成して露出する
- **破壊的 UI 操作**: style 不変の `onPressed` async 化 + AlertDialog による二重確認で、既存 WCAG テストを維持したまま誤タップ耐性を確保する

**派生元**: 2026-04-alarm.md, 2026-04-alarm-3.md, 2026-04-alarm-5.md, 2026-04-alarm-7.md

## P7: 設計レビュー前に「例外系・既存テスト影響・レイアウト予算」を詰める

初稿が正常系だけだとレビュー周回が嵩む。レビュー前に以下を自己チェックする：

- **例外系の4問**: 「ファイル不在は？」「ESCALATED は？」「カウントは before/after？」「DONE は1通りか？」で例外フローを埋める
- **既存テスト影響の洗い出し**: UI 構造変更（ExpansionTile 折りたたみ等）で壊れる既存ウィジェットテストを Implementer に渡す前に特定する
- **レイアウト予算**: 縦スペースが増える変更では「viewport 高さ予算（例: 800×600 で残り 544px）」を明示して階層レイアウトを比較する
